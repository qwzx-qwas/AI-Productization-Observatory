# 不可动摇的主干

自己已经清楚的：

## 项目要解决什么问题
AI Productization Observatory 是一个面向公开网络、可持续运行、偏差可解释的 AI 产品化供给观测系统。

它当前并不是“真实需求探测器”，它只是从另一面来考虑“需求”：

- 最近 30 / 90 天哪些 JTBD 被高频产品化
- 这些结果来自哪些 source item
- 为什么某个 product 被归到某个 taxonomy
- build evidence / clarity / attention 为什么这样判定
- 哪些判断其实不够确定，需要进入 review

## 目标用户是谁
给想要找需求的创作者以另一个角度的灵感，即观测当前已经有什么，来选择那些没能做好或尚未满足的需求

更具体一点，是下面这类人：

- 想找切入点的独立开发者
- 想找细分机会的 AI 创作者
- 想知道某类供给是不是已经过热的人
- 想从“已有供给的结构”反推机会的人

## 核心输入输出是什么
输入是从各平台抓取的数据（目前选择的是product hunt和GitHub）
输出是经过两层分类层后的数据，包含各种数据本身的元信息以及分类结果

再说细一点：

- 输入不是“真实需求”，而是公开网络上可见的供给信号
- 输出也不是“市场结论”，而是结构化后的供给观测结果
- 输出的重点不是给出一个大而全的排名，而是给出可以 drill down 的证据链

## 主要流程是什么
获取筛选后的数据->标准化数据->对数据进行分析分类->输出

中间还有：

- 先保留原始事实
- 再做标准化
- 再做跨 source 的归并(把不同数据源里“其实是同一个供给对 象/产品”的记录进行对齐、去重、合并（同时保留各自来源证据）)
- 再做 taxonomy 和 score 判断
- 最后把结果连回证据

## 成功标准是什么
以高成功率输出可靠的分类后的信息

如果再具体一点：

- 输出稳定，不会今天一个口径明天一个口径
- 分类结果可以回溯到 source item 和 evidence
- 不确定的地方不会假装确定，而是进入 review
- 能稳定看见最近 30 / 90 天里高频被产品化的 JTBD

## 这个系统能回答什么，不能回答什么
能回答的：

- 最近哪些 JTBD 正在被公开地反复产品化
- 某个方向主要是在哪些 source 上出现的
- 某个 product 为什么会被分到这个 taxonomy
- 某个 score 为什么高，为什么低

不能直接回答的：

- 真实世界里哪个需求最强
- 真实市场规模有多大
- 用户真实支付意愿如何
- 高点赞 / 高星标是不是就代表高留存或高商业价值

## 有几条判断轴一定不要混
这个项目里至少有三条轴：

- 需求轴：用户想完成什么工作，也就是 JTBD
- 供给轴：公开网络上已经出现了什么产品化供给
- 构建轴：这些供给有没有 evidence 表明它是 AI-assisted / vibe coding 做出来的

这三条轴不拆开，后面很容易一路混掉：

- 供给频率不等于真实需求强度
- AI-native 不等于 vibe coded
- attention 不等于商业成功
- 公开可见的痛点不等于完整痛点

## 当前阶段先做到哪里
至少在现在这个阶段，目标应该很克制：

- 先只接 Product Hunt 和 GitHub
- 先把最小闭环跑通
- 先把 taxonomy、rubric、schema、prompt contract 定稳
- 先稳定输出最近 30 / 90 天最常见 JTBD 及其证据

暂时不要让它承担这些期待：

- 不做真实需求总量判断
- 不做大而全的数据源扩张
- 不做把所有维度混成一个综合榜
- 不把 dashboard 做成现场推理引擎
