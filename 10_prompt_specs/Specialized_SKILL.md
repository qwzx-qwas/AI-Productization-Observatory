# Specialized_SKILL

本文件只补充 `10_prompt_specs/` 下“阶段执行工作流”这一条窄流程。

若与仓库根目录的 `AGENTS.md` 或 `SKILL.md` 冲突，以全局文档为准；本文件不重写全局工程规范、审查规则或治理要求。

## 1. 适用范围

仅在以下流程中使用本文件：

1. 选取一个阶段 prompt 文档中的一个待执行阶段。
2. 将该阶段内容与统一执行 prompt 一起交给 Codex。
3. 按阶段要求完成实现或修改。
4. 完成后在该阶段末尾追加 `### 已完成`。
5. 根据阶段名、实际改动和仓库现有提交风格生成 commit message 建议。

本文件不授权自动执行 `git commit` 或 `git push`。

## 2. 阶段 prompt 文档选取规则

阶段 prompt 文档只在 `10_prompt_specs/` 中选取，并同时满足以下条件：

- 文件本身包含一个或多个稳定标题：`## 阶段 N：...`
- 该文件用于分阶段执行任务，而不是基础上下文、任务模板、阻塞模板或本地特化 skill

默认排除：

- `10_prompt_specs/00_base_system_context.md`
- `10_prompt_specs/01_task_template.md`
- `10_prompt_specs/02_blocker_response_template.md`
- `10_prompt_specs/Specialized_SKILL.md`

文档选取顺序固定如下：

1. 用户明确指定了某个阶段 prompt 文档时，优先使用该文档。
2. 用户未指定时，只在上述候选文档中选择与当前任务主题直接对应的文档。
3. 主题对应应优先依据文件标题、文档开头的“任务概述”、以及阶段标题中的稳定主题词判断；不要靠模糊猜测选文档。
4. 若仍无法唯一确定目标文档，停止自动推进，返回“找不到唯一对应的阶段 prompt 文档”，请求人工指定。

## 3. 阶段识别与待执行判定

“阶段”必须基于稳定结构识别，不靠语义猜测。

识别规则：

- 仅将形如 `## 阶段 N：...` 的二级标题视为阶段起点
- 某一阶段的范围，从该标题开始，到下一个 `## 阶段 N：...` 之前，或文件结尾
- `### 已完成` 是唯一有效的阶段完成标记
- 只有当 `### 已完成` 出现在该阶段范围内，且位于该阶段内容末尾附近、在下一个阶段标题之前时，才视为该阶段已完成

待执行阶段选择顺序固定如下：

1. 用户明确指定某一阶段时，按用户指定阶段执行，即使前面还有未完成阶段。
2. 用户未指定具体阶段时，默认选择该文档中第一个尚未标记 `### 已完成` 的阶段。
3. 若目标阶段末尾已存在 `### 已完成`，默认视为已完成，不重复执行；只有用户明确要求重做时才允许重做。
4. 若文档中所有阶段都已带有 `### 已完成`，停止自动推进，并返回“无待执行阶段”。

## 4. 执行输入与阅读顺序

本 workflow 的最小输入包括：

- 目标阶段 prompt 文档
- 选中的具体阶段
- 统一执行 prompt
- 全局 `AGENTS.md`
- 全局 `SKILL.md`

执行前的阅读顺序固定如下：

1. 先读仓库根目录的 `AGENTS.md`
2. 再读仓库根目录的 `SKILL.md`
3. 再读统一执行 prompt
4. 最后读取目标阶段 prompt 文档中的目标阶段内容

不要跳过这些上游约束，直接开始实现。

若当前仓库未另行指定统一执行 prompt，默认使用 `10_prompt_specs/00_base_system_context.md`。

## 5. 阶段执行流程

按以下顺序执行，不要交换关键步骤：

1. 确认目标阶段 prompt 文档。
2. 识别目标文档中的当前待执行阶段。
3. 阅读全局 `AGENTS.md` 与全局 `SKILL.md`。
4. 阅读统一执行 prompt。
5. 读取该阶段的完整内容，明确阶段边界、只做/不做、验收标准与参考文件。
6. 将“阶段内容 + 统一执行 prompt + 全局约束”一起交给 Codex 执行。
7. 根据该阶段要求完成实现、修改、验证与结果说明。
8. 回看本次实际改动，确认阶段要求已实际完成，再生成 commit message 建议。
9. 只有在该阶段已经真正完成后，才在该阶段末尾追加 `### 已完成`。
10. 明确保留 `git commit` 与 `git push` 给人工决定和执行。

硬性约束：

- 不要在真正完成前提前写入 `### 已完成`
- 若任务中途阻塞、完成不完整、关键验证未通过，不能写入 `### 已完成`
- 不要因为已经生成 commit message，就自动视为阶段完成

执行要求强调：

请按照以下优先级执行该 prompt：
以文档要求为准；
细节实现参考文档说明；
若文档未覆盖，再参考网上成熟、可靠的实现方案。

## 6. commit message 生成规则

commit message 只能生成建议，不得直接执行 commit。

生成时必须同时结合以下三项：

1. 阶段名或阶段标题
2. 本次实际完成的工作内容
3. 仓库现有 `git log` 中可观察到的 commit message 风格

生成规则：

- 不能机械地只使用阶段标题作为 commit message
- 应先查看近期提交标题，再归纳仓库的现有表述风格
- 输出应简洁、具体、与实际改动相符
- 若阶段范围很大，但本次只完成其中一部分，commit message 必须诚实反映已完成部分，不得夸大为“完成整个阶段”
- 若无法从历史提交中归纳出稳定风格，可退化为“阶段名 + 本次关键动作”的保守写法

## 7. `### 已完成` 回写规则

`### 已完成` 只表示该阶段已经完成。

回写规则固定如下：

1. 追加位置必须在“该阶段内容的末尾”，也就是下一个 `## 阶段 N：...` 之前，或文件结尾处。
2. 若该阶段末尾已经存在 `### 已完成`，默认不重复追加。
3. 若阶段未真正完成，不得提前追加。
4. 若执行结果只是部分完成、待人工确认、或仍存在 blocker，应保持未完成状态，并在正常输出中说明原因。
5. 不要把“已经生成 commit message 建议”误当作“已经完成阶段”。

## 8. 人工边界

以下动作始终保留给人工：

- 是否执行 `git commit`
- 是否执行 `git push`
- 是否采用本文件生成的 commit message 建议
- 是否需要人工改写 commit message

本文件只负责帮助选择阶段、执行阶段、回写完成标记和整理 commit message 建议，不能越过人工确认边界。

## 9. 失败或不确定时的处理

- 找不到阶段 prompt 文档：停止自动推进，明确返回“找不到目标阶段 prompt 文档”，并要求人工指定文件。
- 找不到待执行阶段：若所有阶段都已带有 `### 已完成`，返回“无待执行阶段”；若文档结构不满足 `## 阶段 N：...` 模式，返回“无法按稳定结构识别阶段”。
- 目标阶段已标记 `### 已完成`：默认不重复执行；只有用户明确要求重做时才继续。
- 实际改动不足以支撑“已完成”判断：保留未完成状态，不追加 `### 已完成`，并说明缺失项、验证缺口或 blocker。
- commit message 风格不明确：采用“阶段名 + 本次关键动作”的保守建议写法，不臆造不存在的仓库风格。
